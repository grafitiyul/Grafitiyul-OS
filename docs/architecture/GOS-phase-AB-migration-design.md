# GOS — Phase A + B Migration Design (additive + mirror only)

**Status:** Design proposal only. Nothing applied. No `prisma migrate`, no production change, no
runtime change, no commit.
**Scope:** Phase A (create new foundation tables) + Phase B (one-time read-mirror backfill). No
writer flips. Builds on `GOS-legacy-reconciliation-plan.md` (all blockers locked).
**Last updated:** 2026-06-24

> Safety thesis: Phase A only **adds** structures; Phase B only **copies** data into them. No live
> code reads or writes the new tables, so nothing in production depends on them. Auth stays on
> `AdminUser`; the guide portal stays on `PersonRef`. They remain the sole writers.

---

## 0) Two structural decisions that make A+B safe

1. **DDL and data are separate.** Phase A is **DDL only** (a Prisma migration). Phase B is a
   **separate, idempotent backfill script** run out-of-band — NOT inside a migration. Reason: the
   Railway start command runs `prisma migrate deploy`. If backfill lived inside a migration and
   failed midway, the migration would be marked failed and **block all future deploys**. Keeping
   backfill outside migrations means a backfill hiccup never touches the deploy pipeline.
2. **Phase B is a point-in-time snapshot, not a live mirror.** Because no writer is flipped,
   `AdminUser`/`PersonRef` keep changing and the mirror **will drift** after backfill. That is
   harmless (nothing reads the mirror) and is explicitly a Phase C concern — Phase C re-backfills
   immediately before the flip (or adds dual-write). Stated here so "mirror" is not mistaken for
   "kept in sync."

## 1) Exact tables / models involved

**Phase A — created NEW (empty):**
- `User`, `Role`, `Permission`, `UserRole`, `RolePermission`
- `TeamMember`
- `TeamMemberProfile` — **recommended yes** (mirrors the live PersonRef/PersonProfile split; holds
  the sensitive/bulky fields `bankDetails`, `imageUrl`, `description`, `notes` separately from
  identity).

**Phase A — one ADDITIVE change to a live table (the only live-table touch):**
- `PersonRef` gains a **nullable** `teamMemberId` + FK → `TeamMember`. Additive, no default
  dependency, no runtime reads it. This is the single modification to an existing table in A+B.

**Phase A — migration-aid columns (nullable, removable after legacy retired):**
- `User.legacyAdminUserId` (nullable, unique) — idempotency + traceability back to `AdminUser`.
- `TeamMember.recruitmentExternalId` (already in the draft, unique) — the bridge to
  `PersonRef.externalPersonId`.

**Phase A — prerequisite DDL:**
- `CREATE EXTENSION IF NOT EXISTS citext;` (required before `User`/`TeamMember` citext columns).

**Phase B — read FROM (never modified):** `AdminUser`, `PersonRef`, `PersonProfile`.
**Phase B — written TO (mirror):** `User`, `Role`, `Permission`, `UserRole`, `RolePermission`,
`TeamMember`, `TeamMemberProfile`, and the `PersonRef.teamMemberId` link column (filled where NULL).

**Not involved at all in A+B:** every other Phase-0 model (Contact, Deal, Tour, Communications,
Finance, Forms, Knowledge, Events, etc.). A+B is the **identity+team foundation slice only.**

## 2) Field-by-field mapping

### AdminUser → User (+ Role/UserRole)
| Source (AdminUser) | Target | Rule |
|---|---|---|
| `id` (cuid) | `User.legacyAdminUserId` | idempotency / trace key |
| — | `User.id` | new **UUID v7** (app-generated by the script) |
| `username` | `User.username` | verbatim (citext) — the login handle |
| — | `User.email` | left **NULL** (optional; not the login identity) |
| `passwordHash` | `User.passwordHash` | **verbatim** scrypt `salt:key` |
| `isActive = true` | `User.status` | → `active` |
| `isActive = false` | `User.status` | → `suspended` (NOT `deletedAt`; preserves "deactivated admin doesn't reopen bootstrap") |
| `role` (`admin`) | via `UserRole` → `Role{key:'admin'}` | not a column; an RBAC link row |
| `lastLoginAt` | `User.lastLoginAt` | verbatim |
| `createdAt` / `updatedAt` | `User.createdAt` / `updatedAt` | preserve originals |
| — | `User.displayName` | = `username` (AdminUser has no display name) |

### Role / Permission / RolePermission (seed)
- `Role{key:'admin', name:'Administrator', isSystem:true, archived:false}` — seeded.
- A **coarse** seed permission set with scope `own`/`all` (no `team` scope yet). At minimum a small
  closed set, all granted to `admin` via `RolePermission`. Exact permission keys finalized at
  authoring time; today every admin is full-access, so `admin` maps to the full seeded set.
- `UserRole(userId, roleId=admin)` for each migrated **active** admin User.

### PersonRef / PersonProfile → TeamMember (+ TeamMemberProfile)
| Source | Target | Rule |
|---|---|---|
| `PersonRef.externalPersonId` | `TeamMember.recruitmentExternalId` | **bridge / idempotency key** (unique) |
| — | `TeamMember.id` | new **UUID v7** |
| `PersonRef.displayName` | `TeamMember.displayName` | verbatim |
| `PersonRef.email` | `TeamMember.primaryEmail` | verbatim (citext) |
| `PersonRef.phone` | `TeamMember.primaryPhoneE164` | copy as-is; **flag** non-E.164 in verification (normalization is a later data-quality task, non-blocking) |
| `PersonRef.status` (`active`/`blocked`) | `TeamMember.status` | `active`→`active`, `blocked`→`inactive` |
| — | `TeamMember.memberTypes` | `['guide']` (PersonRef rows are guides) |
| `PersonRef.createdAt` | `TeamMember.createdAt` | preserve |
| `PersonProfile.imageUrl` | `TeamMemberProfile.imageUrl` | verbatim |
| `PersonProfile.description` | `TeamMemberProfile.description` | verbatim |
| `PersonProfile.notes` | `TeamMemberProfile.notes` | verbatim |
| `PersonProfile.bankDetails` | `TeamMemberProfile.bankDetails` | verbatim (sensitive PII) |

**NOT copied (stay on PersonRef — portal/learning runtime):** `portalToken`, `portalEnabled`,
`lifecycleHint`, `identitySource`, `identitySyncedAt`, `accessGrantedAt`, `accessRevokedAt`,
`teamRefId`, and the `FlowTargetPerson` / `Attempt.externalPersonId` references. These remain
PersonRef-owned. `startDate`/`endDate`/`defaultPayRateMinor` on TeamMember are left NULL (no source).

### PersonRef.teamMemberId link
- After TeamMember backfill: `UPDATE PersonRef SET teamMemberId = (matching TeamMember.id)
  WHERE PersonRef.externalPersonId = TeamMember.recruitmentExternalId AND PersonRef.teamMemberId IS NULL`.
- Fills NULLs only → re-runnable.

## 3) Backfill strategy (Phase B)

- A **single idempotent script** (Node, using the Prisma client or parameterized SQL), run manually
  against a target DB. Order:
  1. Seed `Permission` rows, `Role{admin}`, `RolePermission` links.
  2. Upsert `User` from `AdminUser` (key: `legacyAdminUserId`). Generate UUID v7 ids in the script
     (Prisma 5.22 can't; app supplies). Preserve timestamps.
  3. Upsert `UserRole(admin)` for each active-admin User.
  4. Upsert `TeamMember` from `PersonRef` (key: `recruitmentExternalId`).
  5. Upsert `TeamMemberProfile` from `PersonProfile` (key: the TeamMember id).
  6. `UPDATE PersonRef.teamMemberId` where NULL (link step).
- **Small data volumes** (a handful of admins, dozens of guides) → fast, no long locks.
- Reads from `AdminUser`/`PersonRef`/`PersonProfile` are non-locking SELECTs; sources are never
  modified except the additive `PersonRef.teamMemberId` fill.

---

## 4) Idempotency strategy

- **Stable keys, upsert semantics:**
  - `User` keyed on `legacyAdminUserId` (unique) → `ON CONFLICT DO NOTHING`/update.
  - `TeamMember` keyed on `recruitmentExternalId` (unique).
  - `TeamMemberProfile` keyed on its `teamMemberId` PK.
  - `Role`/`Permission` keyed on `key`; `UserRole`/`RolePermission` on their composite PKs.
- **Link fill touches NULLs only** (`WHERE teamMemberId IS NULL`).
- **UUID v7 generation is per-source-row, not per-run:** because ids are written only on first
  insert (upsert no-ops on conflict), re-runs never mint new ids for already-mirrored rows.
- **Convergence:** running the whole script N times yields the same state as running it once. A
  re-run after a partial failure completes the remainder with no duplicates.

## 5) Verification queries (run after Phase B)

1. **Admin parity:** `COUNT(AdminUser WHERE isActive)` == `COUNT(User WHERE legacyAdminUserId IS NOT
   NULL AND status='active' AND has UserRole→admin)`.
2. **Admin completeness:** anti-join `AdminUser` (active) with no matching `User.legacyAdminUserId`
   → **0 rows**.
3. **Password integrity:** every migrated `User.passwordHash` equals its source `AdminUser.passwordHash`;
   no NULLs among migrated rows.
4. **Team parity:** `COUNT(DISTINCT PersonRef.externalPersonId)` == `COUNT(TeamMember WHERE
   recruitmentExternalId IS NOT NULL)`.
5. **Team completeness:** anti-join `PersonRef` with no matching `TeamMember.recruitmentExternalId`
   → **0 rows**.
6. **Link completeness:** `COUNT(PersonRef WHERE teamMemberId IS NULL)` → **0** (or an explained
   list of intentionally-unlinked rows).
7. **No orphans:** `TeamMember.recruitmentExternalId` with no matching `PersonRef` → **0**.
8. **Profile parity:** `COUNT(PersonProfile)` == `COUNT(TeamMemberProfile)`; bankDetails copied
   where present.
9. **Uniqueness holds:** no duplicate `User.username`, no duplicate `TeamMember.recruitmentExternalId`.
10. **Bootstrap dry-run (critical):** compute the union admin-equiv count (legacy + new, per the
    locked §2 condition in the reconciliation plan) → assert **> 0** ⇒ `setupOpen = false`. Setup
    must be locked in every state.
11. **Phone data-quality (non-blocking):** count `TeamMember.primaryPhoneE164` not matching an
    E.164 regex → record for later cleanup.

## 6) Rollback strategy

- A+B are **fully reversible with zero authoritative data loss** — the mirror rows are disposable
  copies; sources are untouched.
- **Down path:** drop FKs → drop `PersonRef.teamMemberId` column → drop new tables in dependency
  order (`RolePermission`, `UserRole`, `TeamMemberProfile`, `TeamMember`, `Permission`, `Role`,
  `User`). `citext` extension can stay (harmless).
- **Backfill rollback:** since backfill is a separate script, "undo" = truncate the new tables (or
  run the down migration). No source data is ever deleted, so nothing to restore.
- Because no runtime reads these objects, rollback has **no behavioral impact** on auth or portal.

---

## 7) Production risks

- **`prisma migrate deploy` auto-runs on Railway start.** The Phase A migration **will apply in
  production on the next deploy** once merged. Mitigation: apply + validate on a scratch DB first;
  keep the migration purely additive and fast.
- **Partial unique indexes / extension:** new tables are empty, so partial-unique index creation
  can't conflict. `CREATE EXTENSION citext` must precede the citext columns — a one-time DB
  privilege requirement; confirm the Railway DB role may create extensions.
- **FK ordering:** `PersonRef.teamMemberId` FK requires `TeamMember` to exist first; sequence the
  DDL accordingly.
- **Backfill-in-migration would block deploys** if it failed — mitigated by keeping backfill OUT of
  migrations (decision §0.1).
- **Mirror drift** between backfill and Phase C — documented as expected (§0.2); harmless until
  flip.
- **Prisma client regen** gains new models; no name collisions (`User`≠`AdminUser`,
  `TeamMember`≠`TeamRef`/`PersonRef`). Existing code paths unchanged.
- **Mixed id strategies** (new UUID v7 vs live cuid) coexist as strings; the script must generate
  UUID v7 explicitly (no DB default in Prisma 5.22).

---

## 8) What must be tested on a scratch DB

1. **DDL apply:** Phase A runs clean on a fresh clone of the prod schema — tables, indexes,
   extension, and the `PersonRef.teamMemberId` column all created without error.
2. **Idempotent backfill:** run the Phase B script **twice**; second run is a no-op; all
   verification queries (§5) pass.
3. **Bootstrap dry-run:** §5.10 returns **locked** in all states tested (both tables populated; new
   table empty; legacy empty).
4. **Behavior unchanged:** smoke-test on scratch that (a) admin login still works against
   `AdminUser`, and (b) a `/p/:token` portal load still resolves via `PersonRef` — proving the
   writers and reads are untouched.
5. **Rollback:** run the down path; confirm the DB returns to its original shape and the app still
   behaves identically.
6. **Tooling:** `prisma validate`, and `prisma migrate diff` between the live schema and the
   proposed schema, reviewed by a human before any prod apply.

---

## 9) Explicitly NOT included in Phase A+B

- **No writer flips:** auth stays on `AdminUser`; portal stays on `PersonRef`.
- **No `auth.js` / runtime / route changes.** No bootstrap-rule implementation (design only).
- **No ongoing sync / triggers / dual-write** — mirror is point-in-time.
- **No retirement** of `AdminUser`, `PersonProfile`, or any legacy table.
- **No `TeamRef` rename**, no `Team` group model, no membership rewiring.
- **No `Contact`** and **none of the other ~30 Phase-0 models** — A+B is the identity+team slice
  only.
- **No email-login change** — username stays the handle.
- **No data deletion, no re-keying** of `FlowTargetPerson` / `Attempt` (they keep resolving via
  PersonRef).
- **No partial-unique enforcement on live tables** (only on the new empty ones).

---

## 10) Exact gate conditions before Phase C

Phase C (flip Team identity to TeamMember; later flip auth to User) may begin **only when ALL**
hold:

1. All §5 verification queries pass against **production data** (read-only, on a prod clone).
2. The backfill script is proven **idempotent** (re-run = no-op) and **reconciled** (counts match)
   on scratch.
3. The **bootstrap union condition** is implemented in code and verified to return **locked** in
   every state (both tables populated, new-empty, legacy-empty) — and **fail-closed** on query
   error.
4. A **drift-handling decision** for the backfill→flip window is made: either re-backfill
   immediately before flip, or add dual-write. (Phase C cannot flip on a stale mirror.)
5. **Rollback rehearsed** successfully on scratch.
6. `TeamMemberProfile` shape finalized (recommended: keep it).
7. Human sign-off that `PersonRef` remains the portal/learning writer until Phase C explicitly
   moves it (identity-only flip first; portal stays).

## Report

**Can Phase A+B be generated as migration files safely?**
- **Phase A: yes**, as a single additive DDL migration (create new tables + `citext` extension +
  the one nullable `PersonRef.teamMemberId` column + indexes). It is safe in content, but note it
  is **not deploy-neutral**: because Railway runs `prisma migrate deploy` on start, the file
  **applies to production on the next deploy after merge**. So "generate the file" effectively
  schedules a production apply — it must be validated on scratch first and merged deliberately.
- **Phase B: should NOT be a migration file.** Generate it as a **separate idempotent backfill
  script**, run out-of-band, so a data hiccup can never block the deploy pipeline.

**Do any live schema conflicts remain?**
- **No model-name collisions** (`User`/`TeamMember` vs existing `AdminUser`/`TeamRef`/`PersonRef`).
- **One additive live-table change:** `PersonRef.teamMemberId` (nullable FK) — non-breaking.
- **Two prerequisites, not conflicts:** the `citext` extension must be enabled (confirm DB-role
  privilege), and UUID v7 ids must be app-generated (no Prisma 5.22 default). Mixed cuid/UUID id
  strategies coexist without issue.

**Is scratch DB validation required before generating files?**
- **Yes — strongly, and before merge.** Precisely because `migrate deploy` auto-applies on deploy,
  the Phase A DDL and the Phase B backfill must be applied and fully verified (§5, §8) on a
  scratch clone first. Only after green verification and human review of `prisma migrate diff`
  should the Phase A migration be merged for production apply.
